# Internationalization (i18n)

How CrossDesk handles user-facing text so it can be translated
into the user's language. Decisions baked from day one because
retrofitting i18n into a project after the fact means refactoring
every print, every message, every label.

Initial languages: **English** (default) and **Polish** (the
project author's native language). Other languages follow as
contributors add them.

WinApps is English-only in `setup.sh`. Cassowary the same. Most
similar VM-management tools are English-first with no translation.
For Polish-speaking and other non-English users this is a real
pain point. Easy win for CrossDesk's positioning.

## Two layers, two tools

CrossDesk has two distinct user-facing surfaces:

1. **CLI / host messages** — output of `crossdesk install`,
   `crossdesk launch`, error messages, log messages.
2. **GUI strings** — Qt6/QML wizard labels, buttons, tooltips,
   status messages.

Each uses a different industry-standard tool. Both produce
translation files contributors can pick up and translate
without touching the source.

### CLI: `gettext` (.po / .mo files)

`gettext` is the freedesktop standard for Linux CLI software.
Workflow:

1. Source code marks translatable strings with `_("...")`.
2. `xgettext` extracts those strings into `crossdesk.pot`
   (template).
3. Translators run `msginit` to create per-language `.po` files
   (`crossdesk.pl.po`, `crossdesk.de.po`, etc.).
4. Translators edit the `.po` files (any text editor, or specialized
   tools like Poedit / Lokalize / Weblate web UI).
5. `msgfmt` compiles `.po` → `.mo` (binary, fast lookup at runtime).
6. Distro package ships `.mo` files at
   `/usr/share/locale/<lang>/LC_MESSAGES/crossdesk.mo`.

Python integration: `gettext` module in stdlib. Set up once at
startup:

```python
import gettext
gettext.bindtextdomain("crossdesk", "/usr/share/locale")
gettext.textdomain("crossdesk")
_ = gettext.gettext

# Then everywhere:
print(_("Installing Windows VM..."))
```

Tooling:

- `pybabel extract -o crossdesk.pot src/`
- `pybabel update -i crossdesk.pot -d locale -l pl`
- `pybabel compile -d locale`

### GUI: Qt's `tr()` macro and `.ts` / `.qm` files

Qt has its own translation system, parallel to gettext. QML uses
`qsTr("...")`:

```qml
Text {
    text: qsTr("Welcome to CrossDesk")
}
```

Source code is parsed by `lupdate` into `.ts` files (XML format,
human-editable). Qt Linguist tool edits them. `lrelease` compiles
to `.qm` (binary) for runtime.

Files:
- `gui/translations/crossdesk_en.ts` — base, machine-extracted.
- `gui/translations/crossdesk_pl.ts` — Polish translations.
- Distro package ships `.qm` files at
  `/usr/share/crossdesk/translations/`.

QML loads them at startup based on `QLocale::system().name()`.

## Why two systems instead of one?

Could we use gettext for both? Technically yes — Qt has gettext
support via `QTranslator` with `gettext` backend. But:

1. Qt Linguist + `.ts` is the muscle memory of every Qt
   contributor; switching to gettext loses contribution velocity.
2. Web translation services (Weblate, Crowdin) handle both `.po`
   and `.ts` natively.
3. The two surfaces (CLI and GUI) have different translation
   contexts; a gettext-only setup would mix them awkwardly.

So: gettext for Python CLI and host messages, Qt's native system
for the GUI. Two source-of-truth files per language.

## Locale detection

At process startup:

```python
import locale
lang = locale.getdefaultlocale()[0]  # e.g. "pl_PL"
```

Qt picks up `QLocale::system()`. Both honor:

- `LANG=pl_PL.UTF-8`
- `LC_MESSAGES=pl_PL.UTF-8`
- `LANGUAGE=pl:en` (gettext fallback chain)

Override via `crossdesk install --language pl` for users on
en-locale systems who want Polish UI (or vice versa).

## Strings we don't translate

Some strings are intentionally untranslated:

- **Log messages.** Logs are diagnostic data for support. We
  log in English so we can debug regardless of user locale.
  User-facing CLI output is translated; log file content is not.
- **Configuration field names.** `RDP_USER`, `enable_audio`,
  field names in TOML files are English. Translation happens at
  the value labels in GUI, not at the schema.
- **Component identifiers.** `host.installer.iso_download_started`
  events stay English so support questions ("show me the event
  log filtered to host.installer") are language-neutral.
- **Error codes.** `EC_VM_OFF=8` etc. stay numeric + English
  symbol. Translation is at the user-facing message level.

This split (English schema, translated UI) is standard
practice and keeps machine-readable data uniform across users.

## Plurals and number formatting

gettext supports plural forms via `ngettext("%d window",
"%d windows", n)`. Polish has more complex plural rules than
English (singular, paucal, plural for 1, 2-4 / 22-24, others).
gettext handles this — translators specify the plural rules in
the `.po` header.

Number formatting (decimal separator, thousands separator) uses
locale-aware formatting:

```python
locale.format_string("%d files", n, grouping=True)
# en_US: "1,234 files"
# de_DE: "1.234 files"
# pl_PL: "1 234 files"
```

## Continuous translation workflow

Recommended setup:

- Source-of-truth `.pot` and `.ts` files in `i18n/` at repo root.
- CI job extracts updated strings on every merge to main, opens
  a PR if changed.
- Optional Weblate integration: hosted translation service that
  reads our repo, lets translators work on a web UI, opens PRs
  with translations.
- Translation contributors are listed in `CONTRIBUTORS.md` with
  the languages they cover.

For now we expect the project author + AI tooling to maintain
English+Polish. Community contributions for other languages
welcome via PR.

## Languages on first release

- **English (en)** — base, all strings authored in English.
- **Polish (pl)** — fully translated by the project author at
  release time.

After first release, expected community languages (none promised,
all community-driven):

- German (de) — large Linux user base.
- French (fr) — significant.
- Spanish (es) — significant.
- Russian (ru) — likely (Linux-heavy community).
- Brazilian Portuguese (pt-BR).
- Japanese (ja).
- Simplified Chinese (zh-CN).

We will not pre-translate these. Community fills in as it cares.

## Sequencing of work

### P0 (foundation, lands with first GUI work)
- `gettext` configured in Python host: `bindtextdomain`,
  `textdomain`, `_()` shortcut available everywhere.
- `_("...")` markers on every user-facing string in the host.
- `qsTr("...")` markers on every QML string in the GUI.
- `pybabel extract` and `lupdate` invocations in `Makefile` /
  `scripts/i18n.sh`.
- `i18n/` directory at repo root with templates and language
  subdirectories.
- Distro package install rules ship `.mo` and `.qm` to standard
  paths.

### P1
- Polish translations complete for all P0-shipped strings.
- CI job: re-extract strings on main; open PR when template
  changes.
- Weblate (or similar) integration evaluated; chosen if
  contribution volume warrants.
- Documentation in `CONTRIBUTING.md` for translators.

### P2
- Plural-form handling audited (most strings don't pluralize; a
  few do — "1 mount active" vs "3 mounts active").
- Locale-aware number/date formatting where it matters
  (file size displays, timestamps in user-visible logs).
- RTL language support evaluated when the first RTL translation
  arrives (Arabic, Hebrew). Qt handles RTL natively; CLI
  generally fine.
